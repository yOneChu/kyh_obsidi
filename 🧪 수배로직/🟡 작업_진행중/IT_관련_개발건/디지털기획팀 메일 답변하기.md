## 1. 가장 현실적인 방법: IT팀이 관리하는 중계 서버 방식

개발자가 API KEY를 직접 받는 게 아니라, **IT팀 서버에만 KEY를 보관**하고 업무 시스템은 그 서버를 호출하는 방식입니다.

```
[사내 업무시스템]   ↓[IT팀 AI 중계 서버]   ↓[OpenAI / Gemini / Claude API]
```

이렇게 하면 개발자는 API KEY를 모릅니다.

장점은 큽니다.

| 항목     | 설명                                 |
| ------ | ---------------------------------- |
| 보안     | API KEY가 개발자 PC나 소스에 노출되지 않음       |
| 비용 통제  | 사용자별 호출량 제한 가능                     |
| 로그 관리  | 누가 어떤 기능을 사용했는지 기록 가능              |
| 데이터 통제 | 외부로 보내면 안 되는 항목 필터링 가능             |
| 모델 교체  | 나중에 Gemini → OpenAI → Claude 변경 가능 |

IT팀에는 이렇게 말하는 게 좋습니다.

> API KEY를 개별 개발자에게 발급받는 방식이 아니라, IT팀에서 관리하는 AI Gateway 또는 중계 API 형태로 구성하고, 업무 시스템에서는 해당 내부 API만 호출하는 방식으로 검토하고 싶습니다.


---

## 2. API KEY 없이 먼저 PoC 하는 방법

정식 연동 전에는 **수동 복사 방식**으로도 PoC가 가능합니다.

```
1. 사내 시스템에서 JSON 조회2. 민감정보 제거3. ChatGPT / Gemini / Claude 웹 화면에 붙여넣기4. 분석 결과 확인5. 효과 검증 후 IT팀에 정식 연동 요청
```

즉, 처음부터 API 연동을 주장하지 않고,

> “AI를 붙였을 때 업무 효율이 있는지 먼저 검증해보겠습니다.”

라고 접근하는 방식입니다.

단, 사내 데이터 반출 정책은 반드시 확인해야 합니다.


---

## 4. API KEY 없이 사용할 수 있는 개발 방식

AI 기능을 실제 API 연동 없이도 화면에 먼저 만들 수 있습니다.

예를 들면:

```
[AI 분석하기] 버튼 클릭→ 사내 API 호출→ JSON 정리→ 프롬프트 자동 생성→ 사용자가 복사해서 ChatGPT에 붙여넣기
```

화면에는 이렇게 보여줍니다.

```
아래 내용을 ChatGPT 또는 Gemini에 붙여넣어 분석하세요.[복사하기]
```

이 방식은 API KEY가 필요 없습니다.

하지만 업무 흐름은 어느 정도 검증할 수 있습니다.

## 추천 순서

제일 현실적인 순서는 이겁니다.

```
1. API KEY 없이 프롬프트 복사 방식으로 효과 검증2. 결과가 괜찮으면 PoC 자료 작성3. IT팀에 API KEY 직접 발급이 아닌 AI Gateway 방식 제안4. 외부 API가 불가하면 사내 LLM 서버 또는 Ollama 검토
```

정리하면, **API KEY를 직접 받는 방식만 고집하지 않는 게 좋습니다.**

가장 설득력 있는 대안은:

> “IT팀이 API KEY를 관리하고, 우리는 사내 중계 API만 호출하겠습니다.”

이 방향입니다.
